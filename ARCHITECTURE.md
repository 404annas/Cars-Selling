## 1. Project Overview

The **Elite Motor Cars** website (`Cars-Selling`) is now a static-exportable Next.js frontend. The dynamic car inventory and admin actions are served by the separate Express backend in `elite-motors-backend`.

The frontend is designed for Hostinger static hosting. It does not depend on Next.js API routes at runtime. Browser requests go directly to the backend configured by `NEXT_PUBLIC_API_BASE_URL`.

### Goal

Complete a dynamic frontend integration where:
- Public pages fetch live car data from the Express API
- The frontend can be exported as static files for Hostinger
- Admin authentication is handled through the backend session endpoints
- The dashboard can list, create, edit, delete, and toggle listings through backend admin endpoints
- Images are stored as URL strings, ready for Cloudinary CDN URLs
- No frontend rebuild is required to update inventory content

```ts
    transmission: string;       // e.g. "Automatic", "Hybrid", "DCT"
    fuel: string;               // e.g. "Petrol", "Hybrid"
    year: string;               // e.g. "2013"
    color: string;              // e.g. "Frozen Blue Matte Metallic"
  };
  highlights: string[];         // Feature bullet points (variable length)
  images: StaticImageData[];    // Local Next.js static images (bundled at build)
  dealer: string;               // Always "Elite Motor Cars"
  dealerLocation: string;       // Always "Sydney, Australia"
```

### 2.2 Key Issues Identified

| Issue | Impact |
|---|---|
| Images are bundled as `StaticImageData` at build time | Cannot add cars without redeploying |
| `price` field is a raw string ("34,999" or "SOLD") | No numeric filtering or sorting possible |
| Duplicate data in `FeaturedCars.tsx` — a separate `cars[]` array mirrors `allCars` | Any update must be done in two places. Very error-prone |
| `id` is a manual integer, non-sequential (0, 1, 2, 3, 4, 5…23, 24, 25…) | Collision risk, cannot auto-increment reliably |
| No categories/tags — only `isSoldCar(price)` boolean logic | No way to flag "Featured", "Hot Deal", etc. |
| `specs.fuel` and `specs.transmission` sometimes say "Hybrid" for both | Ambiguous data — e.g. Audi A4 has `fuel: "Automatic"` |

### 2.3 Pages That Consume Car Data

| Page / Component | Data Source | Notes |
|---|---|---|
| `FeaturedCars.tsx` | Inline `cars[]` static array | Duplicates `allCars` |
| `CarInventoryPage.tsx` | `allCars` from `cars.ts` | Used by `/available-cars`, `/sold-cars`, `/all-cars` |
| `/cars/[id]/page.tsx` | `allCars.find(id)` | Car detail page |
| `/available-cars/[id]/page.tsx` | Re-exports from `/cars/[id]` | Alias route |
| `/sold-cars/[id]/page.tsx` | Re-exports from `/cars/[id]` | Alias route |

## 4. API Endpoints (CRUD)

All API routes live in the separate Express backend: `elite-motors-backend`.

The static frontend calls these endpoints with:

```ts
NEXT_PUBLIC_API_BASE_URL=https://your-backend-domain.com
```

Local development fallback:

```ts
NEXT_PUBLIC_API_BASE_URL=http://localhost:5000
```

### 4.1 Public Car Endpoints (no auth required)

| Method | Endpoint | Description | Query Params |
|---|---|---|---|
| `GET` | `/api/cars` | List all cars with optional filters | `?status=available&tag=featured&sort=price_asc&page=1&limit=12` |
| `GET` | `/api/cars/featured` | Get cars where `isFeatured=true` and `status=available` | — |
| `GET` | `/api/cars/[id]` | Get single car by `_id` or `slug` | — |

**Response shape for `GET /api/cars`:**
```json
{
  "data": [ /* array of Car objects */ ],
  "pagination": { "page": 1, "limit": 12, "total": 37, "totalPages": 4 }
}
```

### 4.2 Protected Dashboard Endpoints (auth required)

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/admin/cars` | List all cars (includes sold) with full data |
| `POST` | `/api/admin/cars` | Create a new car listing |
| `PATCH` | `/api/admin/cars/[id]` | Update an existing car |
| `DELETE` | `/api/admin/cars/[id]` | Delete a car listing |
| `PATCH` | `/api/admin/cars/[id]` | Toggle status or `isFeatured` by sending a partial payload |

> ℹ️ **No image upload API route needed.** Images are uploaded directly from the browser to Cloudinary using the unsigned upload preset. Only the resulting Cloudinary URL is saved to MongoDB via the car POST/PUT endpoints.

### 4.3 Auth Endpoints

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/api/auth/login` | Email + password login, sets session cookie |
| `POST` | `/api/auth/logout` | Clears session cookie |
| `GET` | `/api/auth/me` | Returns current authenticated user (validates cookie) |

## 6. Tech Stack & Justification

| Technology | Role | Why |
|---|---|---|
| **Next.js 16** (App Router) | Static frontend | Already in use. Configured with `output: "export"` for Hostinger static hosting |
| **Express** | Backend API | Runs separately in `elite-motors-backend`; owns API routes, auth, MongoDB access, and CORS |
| **MongoDB** | Database | Flexible schema suits car listings with variable highlights/images arrays. Easy to evolve |
| **Mongoose** | ODM | Used in the Express backend for models, middleware, and validation callbacks |
| **TanStack Query (React Query)** | Client data fetching | Caching, loading states, background refetching, and mutations out of the box |
| **React state + typed hooks** | Dashboard forms | Keeps dashboard logic separated from rendering without adding new dependencies |
| **Zod** | Backend validation | Backend validates car create/update payloads |
| **bcryptjs** | Password hashing | Used by the backend auth controller |
| **Cookie-based sessions** | Auth | Backend sets httpOnly cookies. Frontend checks `/api/auth/me` and redirects client-side |
| **Cloudinary** | Image CDN | Images uploaded directly from browser via **unsigned upload preset** → secure URL stored in MongoDB. Organized by car slug in media library. No API secret needed on frontend |
| **Sonner** | Toast notifications | Already installed in the project |
| **Lucide React** | Icons | Already installed in the project |

### 7.5 Image Upload via Cloudinary (Unsigned — No Server Required)

The `ImageUploader` component uploads images **directly from the browser** to Cloudinary:

```
User selects/drops images in CarForm
  ↓
ImageUploader sends POST to Cloudinary Upload API
  https://api.cloudinary.com/v1_1/{CLOUD_NAME}/image/upload
  Body: { file, upload_preset: "elite-motor-cars-unsigned", folder: "elite-motor-cars/{car-slug}" }
  ↓
Cloudinary returns { secure_url: "https://res.cloudinary.com/..." }
  ↓
URL stored in form state (images: string[])
  ↓
On form submit → URL sent to POST/PUT /api/admin/cars
  ↓
MongoDB stores URL string (not the binary image)
```

**Unsigned upload preset setup (one-time in Cloudinary dashboard):**
1. Go to Cloudinary → Settings → Upload → Upload presets
2. Click "Add upload preset"
3. Set Signing mode: **Unsigned**
4. Set Folder: `elite-motor-cars`
5. Set name: `elite-motor-cars-unsigned`
6. Enable "Auto quality" and "Auto format"
7. Save → add preset name to `.env.local` as `NEXT_PUBLIC_CLOUDINARY_UPLOAD_PRESET`

**Env vars for the frontend uploader:**
```bash
# These are NEXT_PUBLIC so they are safe to expose in the browser
NEXT_PUBLIC_CLOUDINARY_CLOUD_NAME=your_cloud_name
NEXT_PUBLIC_CLOUDINARY_UPLOAD_PRESET=elite-motor-cars-unsigned
```

## 8. Frontend Data Flow

### 8.1 Public Pages

```
FeaturedCars.tsx (homepage)
  ↓ useQuery(['cars', 'featured'])
  ↓ GET {NEXT_PUBLIC_API_BASE_URL}/api/cars/featured
  ↓ Express → MongoDB: Car.find({ isFeatured: true, status: 'available' }).limit(6)
  ↓ Returns JSON array
  ↓ Renders car cards with loading skeleton + error state
```

```
CarInventoryPage.tsx (/available-cars, /sold-cars, /all-cars)
  ↓ useQuery(['cars', { status, page }])
  ↓ GET {NEXT_PUBLIC_API_BASE_URL}/api/cars?status=available&page=1
  ↓ Express → MongoDB: Car.find({ status }).skip().limit()
  ↓ Returns paginated JSON
  ↓ Renders grid with filter tabs + pagination controls
```

```
/cars/[id]/page.tsx (car detail)
  ↓ useQuery(['car', id])
  ↓ GET {NEXT_PUBLIC_API_BASE_URL}/api/cars/:id  (lookups by _id or slug)
  ↓ Express → MongoDB: Car.findById(id) or Car.findOne({ slug })
  ↓ Returns single car JSON
  ↓ Renders full detail with image gallery
```

### 8.2 Dashboard (Admin)

```
Dashboard CRUD flow:
  Admin signs in at /dashboard/login
  → Backend sets httpOnly emc_session cookie
  → /dashboard calls GET /api/auth/me before rendering data
  → Admin fills dashboard car form
  → useAdminDashboard calls POST /api/admin/cars or PATCH /api/admin/cars/:id
  → Express middleware validates session cookie → verifies User in DB
  → Backend validates body with Zod
  → Mongoose saves/updates Car document
  → Dashboard refreshes the admin car table
```

### 8.3 Navigation Performance Strategy

Fast navigation is critical for a car dealership site. Because this frontend is statically exported, runtime data freshness comes from client-side fetching against the Express API.

| Technique | Applied To | Effect |
|---|---|---|
| **`<Link>` prefetching** | All car card links | Next.js prefetches detail pages on hover — instant navigation |
| **Client query cache** | Public car hooks | Avoids refetching the same inventory repeatedly during a session |
| **Loading skeletons** | Featured and inventory grids | Keeps layout stable while backend data loads |
| **Static export** | Hostinger deployment | `next build` emits static files in `out/` |

### 8.4 TanStack Query Cache Strategy

```ts

// GC time: 10 minutes
// Refetch on window focus: false
// Retry: 1 attempt on failure

  defaultOptions: {
    queries: {
      staleTime: 5 * 60 * 1000,

      refetchOnWindowFocus: false,
      retry: 1,
```

## 9. Authentication Architecture

### 9.1 Login Flow

```
1. Admin POSTs { email, password } to /api/auth/login
2. Server: fetch User from MongoDB by email
3. Server: bcrypt.compare(password, user.passwordHash)
4. On success: generate crypto.randomUUID() as session token
5. Save Session { userId, token, expiresAt: +30 days } to MongoDB
6. Set httpOnly, Secure, SameSite=Strict cookie: session_token=<token>
7. Return { user: { id, name, email, role } }
```

### 9.2 Request Authentication (Middleware)

```ts
// src/middleware.ts
export const config = { matcher: ['/dashboard/:path*', '/api/admin/:path*'] }

export async function middleware(request: NextRequest) {
  const token = request.cookies.get('session_token')?.value
  if (!token) return NextResponse.redirect(new URL('/dashboard/login', request.url))
  
  // Validate token against DB (or use a signed JWT for stateless check)
  const session = await validateSessionToken(token)
  if (!session || session.expiresAt < new Date()) {
    return NextResponse.redirect(new URL('/dashboard/login', request.url))
  }
  
  return NextResponse.next()
}
```

### 9.3 Logout Flow

```
1. Admin POSTs to /api/auth/logout
2. Server: Delete Session document from MongoDB
3. Server: Clear cookie (set maxAge=0)
4. Client: Redirect to /dashboard/login
```

## 📌 Notes for Development

> **Images**: Images are uploaded directly from the browser form to Cloudinary via unsigned upload preset. The `ImageUploader` component handles this — no server-side upload route needed. Only the Cloudinary URL is saved to MongoDB.

> **Car-Image Association (Upload Script)**: `scripts/upload-images.ts` uses an explicit `CAR_GROUPS` table that maps each car's slug to its image filenames. Each image is uploaded to `elite-motor-cars/{car-slug}/{image}` in Cloudinary, keeping the media library organized per car. The output `cloudinary-map.json` is then consumed by `seed.ts` to resolve filenames to URLs.

> **Slug Generation**: Auto-generated from car name on save, e.g. `"2013 BMW M5 F10"` → `"2013-bmw-m5-f10"`. Duplicate slugs get a numeric suffix.

> **FeaturedCars**: `isFeatured` is an explicit admin-controlled boolean. On initial seed, `status: 'available'` cars get `isFeatured: true` by default.

> **Price Storage**: Stored as `priceAUD: number | null`. `null` = SOLD. Display formatting (AUD $XX,XXX) happens in the UI layer.
